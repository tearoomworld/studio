Motion · how this team works

Source Marketing runs marathon motion. No outbound; growth comes from consistent content, in-person presence, and word of mouth. The work is daily and compounding — one post Tuesday, one Friday, one event a week, one host conversation a week.

The voice skill source-editorial shapes every Claude API call — content reads as editorial, lowercase, Apartamento-meets-text-from-a-friend. Never SaaS-speak.

Overview

The ledger.

What's working. What isn't. Updated whenever you learn something real from a customer call, a build day, or a moment in the field. Pattern-match across companies.

✓Founder story = trust unlock

"GT + tech + grandma caretaker" lands in every Kindred cold email reply. Use the same shape (credentials + lived experience) for every future company.

✓Single-HTML mocks beat polished pixels

Click-through HTML mocks have closed more conversations than Figma files would. Always lead with something interactive.

✓Audit-before-code saves weeks

The v1→v2 audit pattern catches dormant-data and polymorphism lock-ins before they become migrations.

✓Portal model beats nested tabs

Each company gets its own working portal with its own sidebar. The Studio shell stays for strategy; the portals own the work. Stops the OS becoming a soup of every company's stuff.

✓Different motions can run in parallel

Kindred = sprint (build then sell). Source = marathon (build then sustained content + presence). The work types don't compete for the same hours.

?Lowercase brand voice (Source)

Aesthetic is striking but untested with actual buyers. Validate at first 10 atlanta events before doubling down.

×Vision-without-pilot is procrastination

Writing the "humans-first bible" or designing the talent marketplace before Kindred has a paid customer is a form of avoidance. Note the ideas. Don't build them.

×Copy-pasting structure across companies

Source's sales tool was first built as Kindred's with words swapped. It didn't work. Each company's tabs must come from its actual go-to-market, not from a template.

Plan

Generate a new company.

The intake flow that produces a full asset suite (deck, product mock, marketing site, sales tool if needed) sized to a specific company. Same shape as Kindred and Source, but every detail derived from your answers.

What's real: The questions are real — these are the ones any new company needs answered before assets get generated. What's not yet real: The "generate" button doesn't run a real Claude API yet. Fill out the form, see the meta-prompt, then paste it into Claude Code to actually generate the assets.
1 · What's the company, in one paragraph?
What it does, who it's for, why it matters. Specific over ambitious. ~6 sentences.
2 · What's the go-to-market motion?
The motion determines which Operations sub-tabs auto-generate for this company in your Studio.
Outbound · sprint Defined buyer, you reach out one-by-one. Like Kindred. Operations auto-spawns: Pipeline, Compose, Demos.
Content + presence · marathon You build through IG/TikTok, in-person, word of mouth. Like Source. Operations auto-spawns: Content calendar, Idea generator, Field log, Host scouting.
Paid acquisition · ads Scaled paid channels. Operations auto-spawns: Ad accounts, Creative library, Funnel analytics.
Partnerships · B2B2C One signed partner gets you 100 customers. Operations auto-spawns: Partner pipeline, Joint launches, Co-marketing.
3 · What's the aesthetic?
Color palette, typography, the feel of the brand. Drop in 2-4 moodboard images for visual reference, plus a sentence or two of context.
4 · What's the v2 vision?
What gets added once v1 has traction? Determines dormant tables, reserved user types, feature flags.
5 · What's the wedge — why this, why now?
One sentence on why an incumbent (or absence) leaves room for you specifically. Your founder story stays consistent across all companies (set in Settings).

Settings.

Studio-wide defaults. The locked stack, the founder profile, the patterns every new company starts from.

Studio defaults · locked

Every new company inherits these. Override only when the company specifically needs something different.

Stack
Next.js 15 · Supabase · Vercel · Resend · Anthropic API · Stripe (Connect for marketplaces) · Daily.co or Capacitor as needed
Monorepo
pnpm workspaces. Three apps per company: web, app (or hosts), admin (or dashboard). Shared packages: ui, db, lib.
Database
Postgres on Supabase. RLS on every table. Money in cents (integers, never floats). UUIDs for primary keys.
Auth
Magic-link via Supabase. users.user_type enum from day one — even if v1 has one role, v2 adds more.
Email subdomains
hello@[company].care for sales. notifications.[company].care for transactional. Separate from day one.
Feature flags
Every v2 surface gates behind a flag on the relevant scope (community, city, venue). Default false. Flip per-customer.

Compatibility rules · studio-wide

Applied to every company. Override only with explicit reason.

v2 tables in v1
All defined, empty. Never ALTER TABLE ADD on a live system you can avoid.
Polymorphic columns
Polymorphic from day one. Type columns even when v1 has one value.
Schema additions
Nullable + defaulted. v1 code never writes them.
No premature user surfaces
If a feature requires a user type that doesn't exist yet, it's a v2 feature. Don't sneak in.
Agent budgets
Every Anthropic API call wrapped. Per-endpoint daily cap. Slack alert at 80%.
