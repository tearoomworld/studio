Kindred · Audit

v1 → v2 compatibility audit.

Risks specific to Kindred's product mechanics. Done now so v2 (family-survey pen-pal matching) ships as an additive release with no migration pain.

Contract
The seven Kindred compatibility rules.
Follow these during v1 and v2 ships as an additive release with no migration.
  1. All v2 tables exist in v1. Empty but defined.
  2. Polymorphic columns are polymorphic from day one. sessions.participant_a_type holds 'community' in v1, 'resident' in v2.
  3. Every login is a typed user. users.user_type has 'director' | 'founder' in v1; 'family' added in v2.
  4. Every UI surface checks a feature flag. v2 surfaces ship dark behind community.flags.pen_pals.
  5. Email subdomains separated by purpose. hello@kindred.care for sales; notifications.kindred.care for transactional.
  6. Schema additions are nullable + defaulted. v1 code never writes them.
  7. No resident-level UI in v1. The home is the unit. Residents become first-class in v2.
High
Resident + family tables must exist in v1 (empty), or v2 needs a migration.
Today: v1 doesn't have resident or family UI. Tempting to skip the tables entirely.

Why that breaks v2: If sessions reference "community A vs community B" with no concept of attendees, you have zero behavioral data when v2 wants to suggest pen-pal matches. The tables need to exist so v2 can light up without ALTER TABLE on live data.
Create residents, family_contacts, and session_attendees tables in v1 schema. Hidden from AD UI. Zero rows. v2 starts writing to them when family surveys go out.
High
Auth must be designed for multi-type from day one.
Today: Only ADs log in. Easy to call them "users" and move on.

Why that breaks v2: v2 needs family logins. If v1's users table conflates "human who logs in" with "AD running this community," v2 has to retrofit auth checks and risk leaking community data to family members.
v1 schema: users.user_type column from day one. Allowed: 'director', 'founder'. v2 adds 'family'. Auth middleware checks user_type on every request.
High
Sessions need a stable shape v2 can extend, not replace.
Today: A session is "community A met community B on Tuesday." Group-only.

Why that breaks v2: v2 introduces 1:1 scheduled FaceTimes between two specific residents. If v1's sessions table is hard-wired to "two communities," v2 has to fork or rewrite.
Polymorphic sessions table: session_type ('group' in v1, 'pen_pal' in v2), participant_a_type/id + participant_b_type/id. Same table handles both.
Medium
Communities need a profile field v2 will actually use.
Why it matters: Pen-pal matching works best with community-level context (faith, region, language, vibe). Adding columns later means another migration.
v1 communities table has profile jsonb column. AD onboarding asks 3-4 light questions, stored as JSON.
Medium
Consent + privacy framework needs a stub in v1.
Why it matters: Family surveys collect sensitive info. Need consent records, retention policy, deletion flow. ADs and EDs will ask.
v1 has consent_log table — just AD ToS entries. Same table extends in v2 to family consent, per-category opt-ins.

The v2 user flow

1
AD opts a resident into pen-pal programFrom the director dashboard, AD enables matching on Eleanor and enters family contact info (name + email).
2
Family receives magic-link survey email~10 questions about Eleanor: life history, interests, conversation style. Saves progress.
3
Matching engine generates candidatesFor each opted-in resident, surfaces top 3 candidates with explainable reasons. Rules-based + LLM-assisted.
4
AD picks the matchHuman-in-the-loop. Engine optimizes for good options; AD picks the right one.
5
Pen-pals exchange voice notes, eventually FaceTimeAsync messages flow weekly. AD can schedule a live FaceTime in the dashboard.

Kindred · Agents

Agents + Claude Skills.

Where AI does the real work in Kindred. Copy-paste prompts for Claude Code. Claude Skills that encode design system and codebase conventions. Safety rails to prevent runaway agents.

Trivia + game-content generator

Per-session, themed, age-appropriate. AD picks a theme, API generates 20 questions, cached so we never regenerate. The single most-used AI surface in the product.

Build a trivia question generator in packages/lib/ai/trivia.ts that calls the Anthropic API. Function generateTrivia(theme, count = 20) generates trivia questions for residents 70+. Each question has id, question, choices (4), answer_index, difficulty. Use Claude with structured outputs. Cache in trivia_sets table. Topics: 1950s-60s music, classic films, geography, history. Never sensitive topics.

Session-recap drafter

AD writes 2 lines of notes after each session; agent drafts the warm recap email families can read in v2. AD edits + approves before any send.

Build a session-recap drafter at apps/app/app/(authed)/sessions/[id]/recap/page.tsx. Server component fetches session + director_notes. Claude API drafts a warm, brief recap (3-4 sentences max). Never hallucinate moments not in the notes — if notes are sparse, the recap stays sparse.

Email tactic analyzer + A/B sister

Every cold email gets analyzed against a curated 30-50 tactic library in real time. Suggests improvements. Generates A/B variants you can pick from before sending. The differentiator on the sales side.

In apps/dashboard/app/compose/page.tsx, add a "Tactic analysis" panel. Every 2 seconds (debounced), send draft to /api/email-agent/analyze. Claude API: "Analyze this cold email and identify which tactics from the library it uses. Suggest 2-3 tactics it could add to improve reply rate." Return JSON with used_tactics, suggested_tactics, overall_score. Display in panel.

Reply classifier

Categorizes every inbound reply (yes/no/question/unsubscribe/wrong-person). Surfaces "questions" to your overview as "needs your reply now."

Build a reply classifier in packages/lib/email-agent/classify-reply.ts. Input: original email + reply text. Output JSON: { intent, confidence, summary, suggested_next_step }. Use Claude with JSON mode. Few-shot examples in system prompt. If confidence less than 0.6, queue for manual review.

Lead enrichment agent

Paste "Sunrise Manor, Decatur GA" → agent finds AD name, email, recent programming, regional chain affiliation. Saves 10 minutes per lead.

Weekly digest

Monday 8am email to you: "Your best-performing tactic this week was X. Worst was Y." Vercel Cron triggers it.

kindred-component

Encodes Kindred's design system: peach/serif palette, Fraunces serif, Inter body, warm tone. Triggered for any UI work in the Kindred codebase.

Create a Claude Skill at .claude/skills/kindred-component/SKILL.md that encodes Kindred's design tokens. Include: color palette (--ink, --peach, --peach-deep, --sage, --cream), font usage (Fraunces serif for headlines, Inter sans for body), spacing scale, accessibility defaults (WCAG AA, focus rings), example patterns from the existing HTML mocks. Anti-patterns: no gradients on text, no purple, no "AI-generated startup" aesthetic. Trigger on any UI work.

kindred-supabase

Schema, RLS policies, the seven compatibility rules. Without this, every supabase question requires re-explaining the dormant-tables philosophy.

kindred-voice

Warm tone rules. Banned words ("cutting-edge", "revolutionary", "platform"). The founder voice + when to use the grandma story.

Spam pre-flight check

Before any email leaves your domain: spam-trigger word check, caps ratio, exclamation count, link count, missing unsubscribe. Deterministic, not AI. Fast.

Hallucination guard on the recap drafter

Second-pass verification: "Given these notes, is every claim in the draft supported?" Unsupported phrases highlighted in yellow for AD to remove.

Per-endpoint cost budget

Every Anthropic call wrapped. Per-endpoint daily cap, per-user weekly cap. Slack alert at 80% spend. Solo founders have been bankrupted by runaway agents — non-negotiable.
